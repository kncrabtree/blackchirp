# C++ tree — Agent Guide

The C++ source for the `blackchirp` acquisition application and the
`blackchirp-viewer` companion. This file applies to all work under
`src/` and to test files under `tests/` (the test tree shares the same
conventions). Cross-cutting rules (timeless commits,
no-install-without-consent, subdir nav, build/test commands) are in the
project-root `AGENTS.md`.

## Code style

The canonical reference is
`doc/source/developer_guide/conventions.rst` (rendered at
https://blackchirp.readthedocs.io/page/developer_guide/conventions.html).
Read it before authoring or modifying C++ code. Highlights so the
agent does not omit them at the point of use:

- Classes/structs/enums: `UpperCamelCase`. Functions and variables:
  `lowerCamelCase`. 4-space indentation, spaces only.
- Member variables: `d_` (values), `p_` (raw pointers), `pu_`
  (`std::unique_ptr`), `ps_` (`std::shared_ptr`).
- String literals: `"..."_L1` for ASCII (default), `u"..."_s` for
  non-ASCII or when `QString` is required, `"..."_s` only when the
  call site genuinely needs `QString`. Do not use `QStringLiteral(...)`
  in new code.
- Pass `QString` by `const QString &`. Use `QAnyStringView` for pure
  lookup or pass-through APIs.
- Prefer C++ standard-library data structures over Qt equivalents
  except where the Qt API requires Qt types.

## Settings keys and persistent state

- Settings keys are statically declared in `BC::Key::` (in
  `data/bcglobals.h` / `data/settings/hardwarekeys.h`), `BC::Store::`
  (alongside the data class that owns them), or `BC::CSV::` (in
  `data/storage/blackchirpcsv.h`). Never pass a string literal to
  `SettingsStorage::get` or `set`.
- `SettingsStorage::set` is **protected**; only the owner of a group
  (or its friend) may write. Reads via `get` are public.
- Hardware-default settings are declared via
  `REGISTER_HARDWARE_SETTINGS` (per driver) and
  `REGISTER_HARDWARE_BASE` (per base class), not by ad-hoc
  `setDefault` calls in constructors. The full registration story is
  in `doc/source/developer_guide/hardware_configuration.rst`.

## Logging

All log output goes through the free functions in `data/loghandler.h`:

- `bcLog`, `bcDebug`, `bcWarn`, `bcError`, `bcHighlight` — generic
  log calls.
- `hwLog`, `hwDebug`, `hwWarn`, `hwError` — `HardwareObject` member
  helpers that prepend the device key.

Do not use `qDebug()`, `qWarning()`, or `emit logMessage()` in new
code. Do not call `LogHandler::instance()` directly except in
application-startup connection setup.

Severity selection guide:

| Level | Use for |
|-------|---------|
| `Error` | Failures requiring user action or indicating data-loss risk |
| `Warning` | Automatically-corrected mismatches the user should know about |
| `Normal` | Connection outcomes, experiment milestones, user-initiated state changes |
| `Highlight` | Major milestones (experiment start/end) |
| `Debug` | Hardware lifecycle, configuration loading, protocol details, parameter traces |

## Hardware drivers

New drivers inherit from the appropriate base class in `hardware/` and
live in the same directory. Reuse the existing communication protocols
(`rs232`, `tcp`, `gpib`, `virtual`, `custom`) where possible. The
walkthrough for adding a new driver is in
`doc/source/developer_guide/adding_a_driver.rst`; for adding an entire
hardware type, `doc/source/developer_guide/adding_a_hardware_type.rst`.

## Tests

Tests live under `tests/` (sibling of `src/`, not nested). They use
Qt-Test, are named `tst_<feature>.cpp`, and are registered in
`tests/CMakeLists.txt`. New tests should:

- Use the `tst_` filename prefix and a class name matching the file.
- Cover one feature per file; do not bundle unrelated checks.
- Use the existing test fixtures under `tests/testdata/` rather than
  generating fixture files at test time when the fixture is stable.
- Not assume hardware connectivity — Qt-Test runs in CI without
  hardware and should pass on virtual implementations alone.

The build command is `cmake --build build/tests --target tests`; see
the project-root `AGENTS.md` for the full test invocation.

## Hardware aggregator headers

`hw_h.h`, `hw_base.h`, `hw_impl.h`, `clock_h.h`, `lifhw_h.h` are
auto-generated by CMake from the registered drivers and are gitignored.
Do not edit them by hand; modify the registration in the relevant
driver header instead.

## Vendor libraries

Hardware drivers that depend on vendor SDKs live in
`hardware/optional/<vendor>/` and are wrapped to degrade gracefully
when the SDK is unavailable. The wrapper pattern is documented in
`doc/source/developer_guide/vendor_libraries.rst`.
