# Runtime Hardware Configuration & Plugin System

## Problem Statement

BlackChirp currently requires hardware selection at compile time, creating deployment challenges:

### Current Limitations:
1. **Hardcoded Communication Protocols**: Each HardwareObject hardcodes its protocol (GPIB, RS232, TCP) in the constructor, requiring separate classes for multi-protocol instruments
2. **Compile-time Hardware Selection**: Hardware chosen via CMake configuration at build time (e.g., BC_FTMW_SCOPE=virtual, BC_AWG=M8195A)
3. **Vendor Library Dependencies**: External vendor libraries must be available at compile time, preventing compilation of hardware implementations without installed SDKs
4. **Static Library Linking**: Missing vendor libraries cause compilation failures even when hardware isn't used

### Goal:
Enable runtime hardware configuration while supporting external vendor libraries through dynamic loading, eliminating compilation dependencies and allowing single-binary distribution.

## Solution Strategy

**Core Approach**: Replace compile-time hardware selection with runtime configuration using QLibrary-based vendor library detection. Ship a single BlackChirp binary containing all supported hardware implementations, with runtime availability determined by installed vendor libraries.

**Future Extensibility**: Establish architecture foundations that support Python plugins and vendor-provided extensions without requiring user compilation.

#### Phase 2.5: Configuration User Interface (Week 5-6)

**Objective**: Provide user-friendly interface for runtime hardware configuration with library status feedback. This phase depends on Phase 2.4 ensuring all hardware implementations are compiled in and registered.

**Critical UI Integration Issue Identified:**
MainWindow constructor currently builds hardware-dependent UI elements (menu items, status boxes, control widgets) based on compile-time hardware configuration. With dynamic runtime configuration, this approach is no longer viable. The following components require dynamic UI construction:

- Hardware menu items in `ui->menuHardware`
- Status display boxes for different hardware types (flow controllers, temperature controllers, pressure controllers)
- Hardware-specific control widgets and dialogs
- Connection status indicators

**Required Solution:** Implement dynamic UI construction method that can be called after hardware configuration is determined, replacing the current constructor-time UI building. This method must be able to add/remove UI elements when hardware configuration changes.
